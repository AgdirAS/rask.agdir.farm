A RabbitMQ UI
that doesn't feel like 2009.

Rask is a modern management dashboard for RabbitMQ. Named after Ratatoskr, the Norse messenger squirrel, it replaces the built-in UI with something you'll actually want to use.

Get started View on GitHub ↗
Why Rask

The native UI exposes the broker. It doesn't support the developer.

RabbitMQ's built-in management UI was designed to expose broker internals — not to support developers actually working with it day-to-day. Every feature in Rask exists because developers currently work around the UI. Copying queue names manually. Opening browser devtools to inspect payloads. Keeping a Notepad file of test messages. Refreshing the page to spot trends.

Rask closes that gap by treating the developer workflow as a first-class concern: less clicking, less context switching, less "I wish this just showed me X." These are not exotic features — they are table stakes for any modern developer tool, and the fact that RabbitMQ's UI doesn't have them after 15 years is exactly the gap Rask is built to fill.

At Agdir Drift AS we run RabbitMQ as part of our farming-as-a-service infrastructure and hit these walls daily. Rask is what we wanted to exist.

Features

What's in the box

⚡

Queue management

Sortable, filterable table with live 10-second polling. Critical queue highlighting. Detail drawer with peek, publish, purge, delete, and consumer inspection — all without leaving the page.

🔀

Exchanges & bindings

Full exchange list with type, durability, and auto-delete at a glance. Delete exchanges directly from the UI. Binding overview included.

🌐

Connections & channels

Live view of active connections with per-connection channel breakdown. Close connections from the UI. Dedicated channels page across all connections.

🏗️

Vhosts & users

Create, delete, and configure virtual hosts. Manage per-vhost tracing and permissions. Full user management with tag-based roles.

🚩

Feature flags & limits

View and toggle RabbitMQ feature flags. Configure per-vhost resource limits to cap max connections, channels, or queues.

🌍

Multi-environment switcher

Switch between RabbitMQ instances instantly. Environments are stored as named profiles and activated via a full-screen gateway overlay — no restarts needed.

📊

Overview dashboard

Cluster-level stats, node health indicators, message rates, and connection counts at a glance. Auto-refreshes every 10 seconds.

🌙

Light · dark · system

Full theme support that follows your OS preference by default, with a manual override in the header. No eye strain on either side.

Quick start

Up in three steps

01

Clone and install

Requires Node.js 20+ and pnpm.

git clone https://github.com/agdir/rask
cd rask
pnpm install
02

Configure your connection

Copy the example file and fill in your RabbitMQ details. All values are optional — the defaults work with a local instance using guest/guest.

cp .env.local.example .env.local
03

Run the dev server

Open http://localhost:35672. Port 35672 is chosen to mirror RabbitMQ's own management port (15672) without conflicting with it.

pnpm dev
Configuration

Environment variables

All variables go in .env.local. You can also set them via the Settings page in the UI — useful if you're running a built deployment.

Variable Default Description
RABBITMQ_HOSTlocalhostHostname or IP of the RabbitMQ server
RABBITMQ_MANAGEMENT_PORT15672Management HTTP API port
RABBITMQ_AMQP_PORT5672AMQP protocol port
RABBITMQ_USERguestRabbitMQ username
RABBITMQ_PASSWORDguestRabbitMQ password
RABBITMQ_VHOST/Default virtual host
Architecture

How it works

Rask runs as a Next.js server. The browser never talks to RabbitMQ directly — all requests are proxied through Next.js API routes. AMQP operations (peek, publish, purge) run server-side only via amqplib. 

# Request flow
Browser  →  Next.js API routes  →  RabbitMQ Management HTTP API  (port 15672)
                               →  RabbitMQ AMQP                  (port 5672, server-only)

This means you can deploy Rask behind your existing auth proxy or VPN without exposing the RabbitMQ management port publicly. Rask only needs to reach RabbitMQ from the server side.

Roadmap

What's coming

These are the things we're actively thinking about or building next. Priorities shift — open an issue if something matters to you.

  • soon Azure AD SSO — Single sign-on via Microsoft Entra ID. First auth provider since it's what we use internally at Agdir.
  • soon Docker Compose example — One-command local setup with RabbitMQ + Rask pre-wired.
  • planned Bindings management — Create and delete bindings between exchanges and queues from the UI.
  • planned Standalone publish page — Publish messages to any exchange with routing key and headers, independent of queue context.
  • planned Additional SSO providers — Google Workspace, GitHub, and generic OIDC are on the list after Azure.
